## Problem Statement

Ouroboros는 ~75 모듈, ~1,300 테스트 규모의 중간 크기 코드베이스다.
실제 프로덕션 환경에서는 이보다 **10~100배 큰 리포지토리**를 탐색해야 하는 상황이 빈번하다:

| Scale | Example | Files | Challenge |
|-------|---------|-------|-----------|
| S (Ouroboros) | ~75 modules | ~200 .py | Single-context 가능 |
| M | Django, FastAPI | ~500-2,000 | 선택적 탐색 필요 |
| L | Kubernetes, Linux kernel | 10,000+ | 구조적 분해 필수 |
| XL | Monorepo (Google-scale) | 100,000+ | 도구 + 전략 + 캐시 필수 |

**핵심 문제**: 단일 Claude 세션의 context window로는 L/XL 규모 리포를 이해할 수 없다.
**해결 방향**: Ouroboros의 AC 분해 + 병렬 실행 패턴을 "코드베이스 탐색"에 적용한 Doc Ops Teammate.

## What is Doc Ops?

**Doc Ops = Documentation as Operations**

코드를 실행하는 게 아니라 **코드를 읽고 이해하고 문서화하는 것**을 체계적 오퍼레이션으로 취급하는 접근법.

```
Traditional:  Developer reads code → writes docs manually
Doc Ops:      Agent decomposes codebase → parallel exploration → structured synthesis → validated output
```

### Core Principles

1. **Decompose Before Explore** — 전체를 읽지 않는다. 구조를 먼저 파악하고, 독립 단위로 분해한 뒤 탐색한다.
2. **Explore in Parallel** — 독립적인 모듈/패키지를 동시에 탐색한다.
3. **Synthesize Upward** — 하위 탐색 결과를 상위 레벨로 합성한다 (Bottom-up).
4. **Cross-Reference** — 모듈 간 의존성, import 관계, 공유 타입을 교차 검증한다.
5. **Resumable** — 중단/재개 가능. 이미 탐색한 부분은 다시 읽지 않는다.

## Architecture: The Doc Ops Teammate

### Design: Claude Code Team Structure

```
Team Lead (User's Claude Code session)
    │
    ├── Doc Ops Coordinator (general-purpose teammate)
    │       │
    │       ├── Explorer A (Explore subagent) ─── Module Group 1
    │       ├── Explorer B (Explore subagent) ─── Module Group 2
    │       ├── Explorer C (Explore subagent) ─── Module Group 3
    │       │
    │       └── Synthesizer (general-purpose) ─── Cross-reference + Final docs
    │
    └── TaskList (shared coordination)
```

### Phase 0: Structural Reconnaissance

**목적**: Context window 소모 없이 코드베이스의 골격을 파악한다.

```bash
# Step 1: Directory tree (cheap, fast)
find . -type f -name "*.py" | head -500      # File listing
find . -type d -maxdepth 3                   # Top 3 levels of directories

# Step 2: Entry points
grep -r "def main\|if __name__\|app = \|cli" --include="*.py" -l

# Step 3: Package structure
find . -name "__init__.py" -exec grep -l "from\|import" {} \;

# Step 4: Dependency graph (imports)
grep -rn "^from \|^import " --include="*.py" | head -1000
```

**Output**: `exploration-map.yaml` — 코드베이스의 모듈 그래프와 탐색 우선순위

```yaml
# exploration-map.yaml (generated by Phase 0)
repo:
  name: target-repo
  total_files: 2847
  languages: [python, typescript]

modules:
  - path: src/core/
    files: 42
    priority: 1          # Foundation — explore first
    depends_on: []

  - path: src/api/
    files: 128
    priority: 2
    depends_on: [src/core/]

  - path: src/workers/
    files: 89
    priority: 2          # Same level — can parallelize with api/
    depends_on: [src/core/]

  - path: src/ui/
    files: 340
    priority: 3
    depends_on: [src/api/, src/core/]
```

### Phase 1: Dependency-Aware Decomposition

Ouroboros의 **Kahn's Algorithm (topological sort)** 을 모듈 탐색에 적용:

```
Level 0 (no deps):  src/core/         ← Explore first
Level 1 (deps: L0): src/api/, src/workers/  ← Explore in parallel
Level 2 (deps: L1): src/ui/, src/cli/      ← Explore in parallel
Level 3 (deps: L2): src/integration/        ← Explore last
```

**Ouroboros 패턴 매핑**:

| Ouroboros Concept | Doc Ops Equivalent |
|---|---|
| Seed (goal + AC) | Exploration Spec (scope + questions) |
| AC Tree | Module Tree (dependency-sorted) |
| ParallelACExecutor | Parallel Module Explorer |
| LevelCoordinator | Cross-Reference Validator |
| ExecutionStrategy | Exploration Strategy (depth/breadth/targeted) |
| EventStore | Exploration Journal (what was read, what was found) |

### Phase 2: Parallel Exploration

각 모듈에 대해 **Explore 에이전트**를 병렬로 spawn:

```python
# Pseudocode: What the Coordinator does
for level in topologically_sorted_levels:
    tasks = []
    for module in level.modules:
        task = TaskCreate(
            subject=f"Explore {module.path}",
            description=f"""
            Explore {module.path} and produce:
            1. Module purpose (1-2 sentences)
            2. Key abstractions (classes, protocols, types)
            3. Public API surface (exported functions/classes)
            4. Internal patterns (error handling, state management)
            5. Dependencies (what it imports, what imports it)
            6. Gotchas (non-obvious behavior, magic, implicit contracts)
            """,
        )
        tasks.append(task)

    # Execute all tasks in this level concurrently
    await parallel_execute(tasks)

    # After level completes: cross-reference
    cross_ref = validate_cross_references(level.results)
```

### Phase 3: Bottom-Up Synthesis

```
Individual Module Docs  →  Package Summaries  →  Architecture Overview  →  Final Handoff Doc
     (L0, L1, L2)            (group by pkg)        (cross-cutting)         (human-readable)
```

**Synthesis 에이전트의 역할**:
1. 모듈별 탐색 결과를 읽는다
2. 교차 참조를 검증한다 (A가 B를 import한다면, B의 문서에도 반영)
3. 아키텍처 패턴을 추출한다 (반복되는 패턴, 공유 추상화)
4. 최종 문서를 생성한다

## Implementation Plan: Claude Code Teammate Agent

### Option A: `.claude/agents/doc-ops-explorer.md` (Recommended Start)

단일 에이전트 정의로 시작. Claude Code의 `Task` tool로 팀을 구성.

```markdown
# .claude/agents/doc-ops-explorer.md

## Role
You are a Doc Ops Explorer — a systematic codebase analyst.
You explore codebases methodically, producing structured documentation.

## Process
1. RECON: Run structural reconnaissance (directory tree, entry points, imports)
2. MAP: Build exploration-map.yaml with module graph and priorities
3. DECOMPOSE: Topologically sort modules by dependency
4. EXPLORE: For each level, explore modules (delegate to Explore subagents when possible)
5. SYNTHESIZE: Merge findings into cohesive documentation
6. VALIDATE: Cross-reference imports, types, and interfaces

## Output Format
For each module, produce:
- **Purpose**: 1-2 sentence summary
- **Key Types**: Classes, Protocols, TypeAliases with brief descriptions
- **Public API**: Exported functions with signatures
- **Patterns**: Design patterns used (Result type, event sourcing, etc.)
- **Dependencies**: Imports from and imported by
- **Notes**: Non-obvious behavior, gotchas, implicit contracts

## Constraints
- Never modify target repository files
- Read-only exploration
- Prefer Glob/Grep over Bash for file discovery
- Limit individual file reads to what's necessary (skip test files in first pass)
- Track what you've explored to enable resumption
```

### Option B: Full Team with Coordinator (For L/XL Repos)

```
TeamCreate("doc-ops-team")
    │
    ├── Task("coordinator", subagent_type="general-purpose")
    │     - Runs Phase 0 (recon)
    │     - Creates exploration-map.yaml
    │     - Creates tasks for each module group
    │     - Assigns to explorer teammates
    │
    ├── Task("explorer-core", subagent_type="Explore")
    │     - Explores foundation modules
    │
    ├── Task("explorer-api", subagent_type="Explore")
    │     - Explores API layer
    │
    └── Task("synthesizer", subagent_type="general-purpose")
          - Reads all exploration results

```

### Option C: Ouroboros-Powered (Future — Self-Referential)

Ouroboros 자체를 사용하여 Doc Ops를 실행:

```yaml
# doc-ops-seed.yaml
goal: "Systematically explore and document repository X"
constraints:
  - "Read-only — do not modify target repository"
  - "Produce structured Markdown documentation"
  - "Cover architecture, patterns, API surface, and gotchas"
acceptance_criteria:
  - "Architecture overview with module dependency graph"
  - "Each module documented with purpose, API, patterns, dependencies"
  - "Cross-reference validation passes (no dangling references)"
  - "Handoff document suitable for onboarding new developers"
task_type: "research"  # Uses ResearchStrategy (Read, Glob, Grep, Bash)
```

이 방식은 Ouroboros의 **전체 6 Phase**를 활용:
- Phase 0: 탐색 범위 명확화 (Socratic interview)
- Phase 1: PAL Router로 비용 최적화 (대부분 Frugal tier로 충분)
- Phase 2: AC Tree로 모듈별 분해 + 병렬 탐색
- Phase 3: 탐색 막힘 시 Resilience (다른 관점으로 접근)
- Phase 4: 문서 품질 평가 (completeness, accuracy)
- Phase 5: 발견된 TODO/추가 탐색 필요 항목 처리

## Exploration Strategies (From Ouroboros `ExecutionStrategy`)

### ResearchStrategy (기본)
```
Tools: Read, Glob, Grep, Bash (read-only commands)
Focus: Broad coverage, API surface mapping
Best for: M-scale repos, initial exploration
```

### DepthFirstStrategy (Deep Dive)
```
Tools: Read, Glob, Grep
Focus: Single module deep analysis — trace execution paths
Best for: Critical modules, debugging-oriented exploration
```

### ArchitectureStrategy (Cross-Cutting)
```
Tools: Read, Glob, Grep, Bash
Focus: Import graphs, shared types, communication patterns
Best for: Understanding how modules connect
```

## Key Technical Decisions

### Q: Why not just `Read` every file?

| Approach | Files Readable | Context Cost | Comprehension |
|----------|---------------|--------------|---------------|
| Read all | ~50-100 files max | Blows context | Surface-level |
| Grep → Read targeted | 500+ files scannable | Efficient | Deep where needed |
| Decompose → Parallel | Unlimited | Distributed | Systematic |

### Q: How does this differ from `tree` + `grep`?

Raw tools give you data. Doc Ops gives you **understanding**:
- `tree` → file listing. Doc Ops → module dependency graph with priorities.
- `grep "class"` → class names. Doc Ops → class purposes, relationships, patterns.
- `cat file.py` → raw code. Doc Ops → "this module implements X pattern for Y reason."

### Q: When to use Option A vs B vs C?

| Repo Scale | Recommendation | Reason |
|-----------|----------------|--------|
| S (~200 files) | Option A (single agent) | Overkill to parallelize |
| M (~2,000 files) | Option A with targeted scope | Focus on key modules |
| L (~10,000 files) | Option B (full team) | Parallel exploration essential |
| XL (~100,000+ files) | Option C (Ouroboros-powered) | Need full pipeline: decomposition, resilience, evaluation |
