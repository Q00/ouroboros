# RFC: Plugin Tool-Call Hook Contract

## Status

Proposed Wave 2 documentation slice for #939, tracked from #1142 PR C-docs.
This document is intentionally **docs-only**: it defines the target contract for
a future PR F dispatch implementation and does not introduce a manifest schema,
runtime dispatcher, permission check, or audit-event emitter by itself.

## Relationship to the existing plugin RFC

The base UserLevel plugin contract lives in
[`docs/rfc/userlevel-plugins.md`](userlevel-plugins.md). That RFC already pins
lifecycle hook vocabulary and the v0.3 terminal-observability precedent shipped
through #1137: observability hooks are fail-open, bounded, and audited without
letting plugin code mask the original command outcome.

Tool-call hooks extend the same firewall model to individual tool invocations.
They are deferred until PR F because dispatch requires a v0.4 manifest/schema
surface and a concrete tool-call interception point. Until PR F lands, this file
is the review target for payload shape, permission boundaries, failure policy,
and audit event names only.

## Hook kinds

| Hook | Purpose | Side-effect class | Eligible failure policy |
| --- | --- | --- | --- |
| `before_tool_call` | Runs after the harness has identified a plugin-mediated tool call but before the tool is allowed to execute. | Policy / optional bounded mutation gate. | `fail_closed` when declared with `plugin:tool:intercept`; `fail_open` when declared with `plugin:tool:observe`. |
| `after_tool_call` | Runs after the tool returns or fails, with bounded result metadata. | Observation only. | `fail_open` only. |

`before_tool_call` is the only hook in this contract that may become
fail-closed. `after_tool_call` must never block, replace, or reinterpret the
already-observed tool result.

## Payload contract

All payloads are JSON objects with deterministic keys. Implementations must
bound payload size before invoking plugin code and must redact or replace secrets
with references or hashes.

### Shared fields

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `schema_version` | string | yes | Future PR F introduces `plugin_schema_version: "0.4"`; hook payloads use the same major/minor compatibility window. |
| `invocation_id` | string | yes | Firewall-owned plugin command invocation identifier. |
| `tool_call_id` | string | yes | Stable identifier for this attempted tool call within the invocation. |
| `plugin_id` | string | yes | Manifest plugin name or canonical first-party plugin id. |
| `command` | string | yes | Plugin command that caused the tool call. |
| `tool_name` | string | yes | Harness-visible tool name, for example `terminal`, `read_file`, or an MCP tool id. |
| `tool_category` | string | no | Coarse category such as `filesystem`, `network`, `process`, `mcp`, or `unknown`. |
| `permission_scope` | string | yes | The scope that authorizes this hook path: `plugin:tool:intercept` or `plugin:tool:observe`. |
| `started_at` | string | yes | RFC 3339 timestamp generated by the firewall. |
| `correlation` | object | no | Bounded ids such as run id, session id, workflow id, step id, or AC id. |

### `before_tool_call` fields

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `redacted_args` | object | yes | Argument preview after the firewall removes secrets and truncates large values. |
| `args_digest` | string | yes | `sha256:` digest over the canonical unredacted argument JSON, used for correlation without storing raw input. |
| `declared_risk` | string | no | Harness-supplied risk label when available, for example `read`, `write`, `network`, or `destructive`. |
| `default_decision` | string | yes | The firewall decision before plugin advice, usually `allow`. |

Allowed plugin responses for a fail-closed `before_tool_call` hook are:

| Response field | Meaning |
| --- | --- |
| `decision: "allow"` | Continue with the original tool call. |
| `decision: "block"` | Block the tool call and emit `plugin.tool.intercept.blocked`. |
| `decision: "replace_args"` | Replace arguments only if PR F explicitly enables bounded mutation for the tool category; otherwise treat as `block` with a contract-violation reason. |
| `reason` | Human-readable, bounded explanation recorded in audit metadata. |

For observe-only declarations, any response other than advisory metadata is
ignored and the original tool call proceeds.

### `after_tool_call` fields

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `status` | string | yes | `success`, `failed`, `cancelled`, or `blocked`. |
| `duration_ms` | integer | no | Non-negative elapsed time if measured. |
| `output_ref` | string | no | Bounded artifact or log reference for large output. |
| `output_digest` | string | no | `sha256:` digest for raw output when a stable byte representation exists. |
| `error_type` | string | no | Bounded exception/category name for failed calls. |
| `error_preview` | string | no | Redacted and truncated error text. |

`after_tool_call` never receives unbounded stdout, stderr, file contents, or
provider responses inline. Large material must be represented by `output_ref`, a
digest, or both.

## Permission scopes

| Scope | Class | Required behavior |
| --- | --- | --- |
| `plugin:tool:intercept` | Mutating / policy | May declare `before_tool_call` with `fail_closed`. Requires explicit trust before dispatch. A denied or missing trust grant blocks the hook path and, when the hook is required for the command, blocks the tool call. |
| `plugin:tool:observe` | Observation | May declare `before_tool_call` or `after_tool_call` with `fail_open`. Missing trust must prevent the hook from running, but must not block the original tool call solely because observation was unavailable. |

Scopes use the colon-delimited manifest grammar from
`docs/rfc/userlevel-plugins.md`. Dot-delimited aliases such as
`plugin.tool.observe` are not valid.

## Failure policy

- `plugin:tool:intercept` + `before_tool_call` may use `fail_closed` because it
  is an authority-bearing hook.
- `plugin:tool:observe` hooks must use `fail_open`; timeout, crash, invalid
  response, or malformed advisory output is audited and ignored.
- `after_tool_call` must always be `fail_open`, including when a plugin attempts
  to declare a stricter policy.
- Hook retries, if PR F adds any, must be bounded and must preserve the original
  tool call idempotency contract.
- A terminal observation hook must not mask the original tool error, success, or
  cancellation result. This mirrors the #1137 lifecycle-observability precedent.

## Audit events

Future PR F must use these event names if it implements this contract:

| Event | When emitted | Minimum outcome fields |
| --- | --- | --- |
| `plugin.tool.intercept.requested` | Before running a `before_tool_call` hook. | `tool_call_id`, `tool_name`, `permission_scope`, `failure_policy`. |
| `plugin.tool.intercept.completed` | A `before_tool_call` hook returned valid advice. | `decision`, bounded `reason`, elapsed time. |
| `plugin.tool.intercept.blocked` | The firewall blocked a tool call because a fail-closed hook denied it, failed closed, or violated the contract. | `block_reason`, original `tool_call_id`, optional hook id. |

Observation-only `after_tool_call` execution may continue to use the generic
`plugin.hook.invoked` / `plugin.hook.completed` / `plugin.hook.failed` vocabulary
unless PR F explicitly vendors more specific `plugin.tool.observe.*` events.
Core must not emit any new event name until the upstream plugin audit schema and
the vendored core copy both allow that name.

## PR F migration plan

PR F is the future dispatch implementation slice. It should introduce, in one
reviewable step or a deliberately stacked set of PRs:

1. `plugin_schema_version: "0.4"` manifest/schema support for tool-call hook
   declarations.
2. Loader validation for hook kind, permission scope, timeout, and failure
   policy combinations described above.
3. Firewall dispatch for `before_tool_call` / `after_tool_call` at a concrete
   harness interception point.
4. Audit schema vendoring for the event names in this document before runtime
   emission.
5. Negative tests proving observe-only hooks remain fail-open and intercept
   hooks cannot run without explicit trust.

PR F must not infer broader plugin state, artifact write, or all-event
subscription hooks from this contract. Those remain separate design surfaces.
